# lies-above.github.io

URL redirection service for the book *Lies Above*, hosted on GitHub Pages.

Each footnote QR code points at a stable path like `/v1/fn5/1`. The site serves
one page per reference that shows the visitor where they are going and forwards
them to the cited source. `redirects.json` is the single source of truth,
`gen.sh` turns it into the whole site, and a GitHub Actions workflow runs
`gen.sh` on every push and deploys the result to Pages.

## Source of truth

`redirects.json` is a flat map keyed by the bare reference ID, each value
`{ "url", "description" }`.

```json
{ "fn5.1": { "url": "https://example.org/source", "description": "..." } }
```

The key `fn5.1` is prefixed with the edition (`v1.`) to form the lookup ID
`v1.fn5.1`, served at `/v1/fn5/1/`, so the printed QR URL `.../v1/fn5/1`
resolves to it. Multi-source footnotes use dotted keys (`fn1026.1`, `fn1026.2`).

## How a redirect page works

Each generated page is self-contained (no external resources, all CSS and JS
inline, system fonts only) and shows the destination before forwarding:

- `<link rel="canonical">` declares the destination to crawlers.
- An inline script shows a short styled notice with the reference ID and the
  destination URL, then forwards with `location.replace(...)` (no back-button
  trap). A visitor can tick "always go straight to sources" to skip the notice
  on later scans. That choice is saved in `localStorage` (key `la:skip`), and an
  On/Off toggle in the home and references footers turns it back off.
- A `<noscript>` meta refresh forwards visitors who have JavaScript disabled.

A request to `/?id=v1.fn5.1` is rewritten by the root page to `/v1/fn5/1/`.
Unknown references fall through to the branded `404.html`.

The reference index at `/references/` lists every entry with its destination URL
in plain text and a client-side search box, so a reader can see where a code
leads before scanning it.

## Build and deploy

The generated tree is **not** committed. GitHub Actions builds and deploys it:

1. One time: GitHub repo → Settings → Pages → Build and deployment → Source →
   **GitHub Actions**.
2. On every push to `main`, `.github/workflows/deploy.yml` runs `gen.sh` and
   publishes the site to Pages.

The repo holds only the inputs: `redirects.json`, `gen.sh`, the hand-maintained
pages, and the assets (`CNAME`, `favicon.png`).

To preview locally:

```sh
sh gen.sh                 # writes index.html, 404.html, references/, and v1/
                          # into the current directory
python3 -m http.server    # then open http://localhost:8000/v1/fn1/
```

The generated paths (`/index.html`, `/404.html`, `/references/`, `/v1/`) are
gitignored, so a local run never dirties the tree.

**Dependency:** [`jq`](https://jqlang.github.io/jq/), pre-installed on the CI
runner. It parses the JSON and escapes every destination URL: `@html` for the
HTML-attribute slots, `@json` for the JavaScript string. Do not parse
`redirects.json` with `grep` or `sed`; that loses the escaping guarantees.

## Hand-maintained pages

A few entries in `redirects.json` point back at this same host (for example a
footnote citing `https://liesabove.app/bibliography`). Those paths are real
files, edited by hand and copied into the build as-is, not generated by `gen.sh`:

- `bibliography/index.html`
- `index/index.html`
- `privacy/index.html`
- `ai/index.html`

They share the landing page's minimal branding. The bare path `/index` resolves
to the root landing page; the placeholder is served at `/index/`. When you add a
new hand-maintained page, also add it to the copy step in
`.github/workflows/deploy.yml`.
